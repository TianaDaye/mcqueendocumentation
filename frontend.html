Navigation and User Authentication

This logic is complex, and we wanted to fully document it out.

BrowserRouter, Routes, Outlet, and Layout:


Layout.js

  • Contains conditional rendering

  • Each layout contains its respective nav bar, and an Outlet tag. The component defined in the associated route is ‘plugged in’ to the outlet tag.

  • The Route components are nested within a Layout component that manages the overall layout, including the header and is conditioned on the user's authentication and role.


UseAuth Hook - User Context

  • The useAuth hook accesses the authentication context, which provides user and logout functions.

  • This hook refreshes and re authenticates the user when necessary


Navigation.js

  • Called by App.js, nested within BrowserRouter, using React Router

  • It is the central routing mechanism for the entire application.

  • It ensures that the application can dynamically render different components based on the user's status and interactions with the app.

  • It handles:

    • User authentication

    • Role-based access control

    • Session management

    • User inactivity

  • UseNavigate controls navigation, and useLocation controls the current URL path.

  • A state employeeRoleNum stores the employee's role from session storage.

  • It can set an employee's role number (setEmployeeRoleNum) which affects the accessibility of various routes within the app based on their role.

  • An InactivityDetector component is used to log out the user after a specified period of inactivity.

  • Uses useEffect to update and recall the last location visited that isn't related to a reservation process. This is used for redirecting the user back to their last non-reservation-related page..

    • This is to ensure the page location persists on an event such as a page refresh

  • A set of Route components defines the URL paths and their corresponding React components.

    • These include Public Routes: Home, Login, Email Verification, Registration, and informational pages (FAQ, Contact, About Us).

    • Protected Routes: These routes require authentication and are wrapped in a ProtectedRoute component. They include user profiles and reservation-related pages.

    • Employee Routes: Wrapped in an EmployeeRoute component, these routes are accessible only to users with specific employee roles. They include dashboards and management tools for different aspects of the business, such as fleet, employees, customer approvals, etc.

  • Based on employeeRoleNum, certain routes will render either the intended component or a fallback Unauthorized component if the user's role does not grant them access.

  • If users access a path they have previously visited, they are redirected back to that path, ensuring they return to the last relevant context.

  • Defines a state for if the user is logged in, and a function to change this state (isLoggedIn).

  • Default route calls the Layout component with the isLoggedIn function as a property

    • This function will be used to pass the isLoggedIn state from child component (Login) to the parent component (Navigation)

Front End File Structure

* BOLDED - Directory


Assets

  • Images used across the site - these were provided by the client.

Customer-Pages

  • About-Us.js

    • Displays the information provided by the client. Contains a call to action to sign up.

  • Contact.js

    • Uses the ContactForm component to allow users to submit any inquiries pertaining to vehicles, general questions, general feedback and website feedback.

    • Sends data to the API in order to display the inquiry to customer service.

  • Faq.js

    • Displays the FAQ provided by the clients in an MUI Accordion component

    • Utilizes the ContactForm again in case users have additional questions.

      • MUI Accordion

  • FormConfirmation.js

    • Provides confirmation after a user creates an account successfully. They are automatically redirected here after their account information has been successfully posted to the API.

  • Homepage.js

    • Call to action button to redirect users to signup.

    • Displays steps to creating an account and reserving a Gyrocar using the HowItWorks component.

    • Displays promotional content provided by the clients

  • Login.js

    • Contains a useEffect hook to determine where the user should be directed to after login

      • Uses the UseAuth() user to test is user is an employee or not

      • If the user is an employee, they navigate to their dashboard

      • If the user is a customer, they navigate to the reservation page

      • Sets the sessionStorage item ‘role’ to the user role

    • Has a form using MUI Textfields to collect the username and password.

    • On submit, the username and password are posted to the API and validated. If the combination is incorrect, the users will see an error message informing them to try again.

  • Profile.js

    • Checks whether the user is an employee or a customer, and renders the correct UI

    • Customer Profile

      • The user ID, Name, Email, and Phone Number are accessed from the UseAuth() hook

      • This information is displayed to the left hand side of the page, and the email and phone number can be edited.

      • An API call is made to access all reservations tied to the customer's ID. These are displayed to the right hand side of the page, and the upcoming reservations are tagged with an ‘Upcoming’ flag.

    • Employee Profile

      • Employee information is accessed via the useAuth() hook, and they can view their Name, Employee Number, Email, and Role.

  • Register.js

    • This is the gateway to the sign up form. The Form component is wrapped in the FormProvider component from FormContext.

  • Reservation.js

    • This file contains the form to search for a reservation. Select Menus are used to select start and end stations, and Date Time Pickers are used for start and end date selections.

    • Input is validated by ensuring the end date is after the start date, and that both dates are in the future.

    • The MapResults component switches between rendering just the MapComponent, or the MapComponent with search results for available cars. If the search button is hit, and the input is validated, the results section is rendered.

    • The search query is sent to the API, and the response returns available cars based on the query. This data is passed down from the Reservation page to the MapResults component as a parameter, in order to render the results.

    • When the Result card is clicked, the selected reservation data is passed through the URL parameters using UseNavigate from React-Router-Dom to the ReservationDetails page.

  • ReservationConfirmation.js

    • After the reservation is processed and there is a 200 status from the API after posting the reservation, the user is navigated to the ReservationConfirmation page using UseNavigate.

  • ReservationDetails.js

    • This page takes in the selected reservation details from the URL parameters using useLocation from React-Router-Dom.

    • The details are displayed and the user has a select menu to choose their card for payment. The cards are displayed by making an API call for the card ID that is associated with Stripe.

  • Unauthorized.js

    • This page shows the 403 error if any user tries to access a URL that they are not authorized to view. The Navigation.js, ProtectedRoute.js, and EmployeeRoute.js handle this logic.

  • VerifyEmailPage.js

    • It verifies a user's email through a token passed via the URL query parameters.

    • The query parameters are accessed using new URLSearchParams(location.search), and the email verification token is retrieved.

    • If no token is found, an alert notifies the user, and they are redirected to the home page (navigate('/')).

    • If a token is provided, an API request is made using axios.get to a constructed URL containing the token as a query parameter.

    • If the response status is 200 (HTTP OK), it implies the email was successfully verified, and the user is redirected to a confirmation page (navigate('/registration-confirmation')).

    • If the response status is not 200, an alert with an error message derived from the response is shown, followed by a redirection to the home page.

    • If the API request fails (e.g., due to network issues), an error is logged, an error message is displayed, and the user is redirected to the home page.

    • The component renders a simple "Loading..." message while the email verification process is ongoing.

Employee-dashboards

  • Full-Dashboard.js

    • Uses the employee role id from useAuth() hook to determine which dashboard to show with conditional rendering and if statements.

      • Admin = 1, Customer Service = 2, Mechanic = 3, Manager = 4

    • Displays a DashCard for each piece of information. These cards are wrapped in a Link component from React-Router-Dom to navigate the user to the correct details page.

  • MetricsDashboard.js

    • Displays MetricsDashCard for both type of metrics - Application and Rental. These cards are also wrapped in a Link component to take the user to the correct URL after clicking on a card.

Employee-Pages

  • CustomerServiceFunctions.js

    • Contains MUI Tabs for each Customer Service page - Approval Requests, Customer Inquiries, Members Info.

    • The URL for this page contains a tab param, which is accessed using the useParams function from React-Router-Dom. This number determines which tab the page should be on depending on the selected DashCard.

  • EmployeeManagement.js

    • This page contains a MUI Table for display all employees.

    • Sorting, searching, and pagination is done on the backend within the API request to getEmployees. When a user sorts, searches, or goes to a new page, a new API request to get the Employees is executed.

    • Admins have the ability to add an Employee, which opens a Dialog Box. Once the Admin fills out the required information, the data is sent to the API to create a new employee.

    • Admins can also edit employees in a dialog box, or delete an employee. They will get a confirmation dialog box before the Employee is fully deleted.

    • Managers can view all employees, but they cannot edit, create, or delete them. This is determined by their role from the useAuth hook

  • FleetManagement.js

    • Stations:

      • UseEffect hook is called to get stations on page load, and will be run every time the number of stations changes

      • Admins can add, delete, or edit stations. Each action opens a MUI Dialog Box for data input or for confirmation.

      • Adding a station - Dialog box opens, and the submit button is disabled until all fields are filled out. When the user submits, the station is added to the database via an API post request.

  • ApplicationMetrics.js

    • It is used to fetch and display application metrics based on certain parameters.

    • A useEffect hook triggers the fetchData function whenever startDate, endDate, statusCode, sortField, or sortOrder changes.

    • The fetchData function makes a POST request using Axios to a specified endpoint with the state variables as parameters. It also includes credentials with the request.

    • If the request is successful, the response data (metrics) is stored in the data state. If there is an error during the fetch, it is logged to the console.

    • A TableContainer houses a Table that displays the fetched data. Each row shows the month, status code, and count from the data array.

    • Descriptions for each status code are provided at the bottom in a Typography component for clarity.

  • RentalMetrics.js

    • It is a self-contained dashboard for visualizing rental metrics data based on user-selected criteria.

    • startDate and endDate are used to store date values for filtering the rental metrics data.

    • sortField determines the field used for sorting the data and is initialized to 'reservationDate'.

    • sortOrder determines the direction of sorting ('asc' for ascending or 'desc' for descending) and is initialized to 'asc'.

    • An effect hook (useEffect) is used to trigger data fetching whenever startDate, endDate, sortField, or sortOrder changes.

    • Inside useEffect, a function fetchData is defined and called immediately. It checks if both startDate and endDate are provided. If so, it makes an HTTP GET request using axios to fetch data from a constructed URL.

    • Upon successful data fetching, the reservations state is updated with the fetched data. If an error occurs, it logs the error to the console.

    • A table displays the fetched rental metrics, sorted and filtered based on user input. Each row shows details for a reservation, including date, station ID, and counts for completed, current, in-progress, and total reservations.

    • A Typography component displays a key for station IDs, mapping them to specific locations.

Employee-Components

  • ApprovalDetails.js

    • The customer ID is taken from the URL using useParams from React-Router-Dom.

    • The customer application is displayed, and the customer service employee can choose to approve the customer, suspend the customer, or save and exit.

    • Approving the customer changes their status code to RDY through a patch request, and verifies their phone and email.

    • Suspending the customer changes their status code to SPD through a patch request

    • Saving and exiting keeps the status code as “PVN”

  • ApprovalRequests.js

    • There are two MUI tables, New Requests and Completed Requests

    • All customers are returned from an API get request, and are mapped through on each table. For the New Requests table, customers are displayed if their statusCode = “PVN” (Pending Verification). For the Completed Requests Table, customers are displayed if their statusCode = “RDY” (Ready to Rent)

    • When the ‘Manage” button is clicked on the New Requests table, the associated customer ID is sent through the URL to ApprovalDetails.js

    • When the “View Details” button is clicked in the Completed table, the associated customer ID is sent through the URL to CustomerDetails.js

  • CustomerDetails.js

    • Displays customer details and their past and upcoming reservations, based on their CustomerID that was passed through the URL params.

  • CustomerInquiries.js

  • DashCard.js

    • Basic component that takes in the display statistic and captions of each piece of information displayed on the dashboard. Serves as a link to the corresponding page.

  • MembersInfo.js

    • Table of all customers, sortable, searchable, and paginated like the EmployeeManagement tables.

    • All members with status codes of “RDY”, “BND”, and “SPD” are displayed. (Ready, Banned, Suspended). Customers pending verification are not on this page, as they are not full members. Banned and Suspended customers are displayed in order to keep them in the system - preventing them from rejoining.

  • MetricsDashCard.js

    • Similar to DashCard, this takes in title and subtitle for the card that is displayed on the Metrics Dashboard.

Hooks

  • InactivityDetector.js

    • This component is tracking events on the screen and setting a timer for inactivity. This timer is cleared during any mouse movement, key stroke, or scroll motion.

    • This is used in Navigation.js to log out a user after a period of inactivity. The user will be redirected to the login.

  • useFormContext.js

    • Very simple react hook to ensure all pages of the sign up form use the FormContext.js logic.

Navigation

  • EmployeeRoute.js

    • If the user is not an employee, or not logged in, this route will redirect them to an Unauthorized screen when they try to access an employee specified route.

    • If they are a logged in employee, they will navigate to the correct screen.

  • ProtectedRoute.js

    • If the user is not logged in, or if they are not a customer, this route will redirect them to an Unauthorized screen when they try to access a user page.

    • If they are a logged in user, they will navigate to the correct screen.

  • Navigation.js

    • This page handles all Routes within the Browser Router.

      • Groups routes by user type (employee versus user), and sets permissions on employee pages based on role. For example, admins and mechanics have access to fleet information, but managers and customer service reps do not. They would be redirected to the Unauthorized screen if they tried to access that URL.

    • Sets the timer for the InactivityDetector, and logs out the user when the screen becomes inactive for the set threshold.

  • Layout.js

    • Conditionally renders the different global navigations based on if the user is logged in, and if they are an employee or a customer.

    • Contains the Outlet tag

Signup-Components

  • Form.js

    • Create a shell of the form, which contains the FormInputs (CustomerInfo, DriversLicenseInfo, CreditCardInfo).

    • Handles submit of the data to the API to create the customer

      • Sends requests to the API for all three pages on submit. It waits until all pages are filled out and validated before sending data.

    • Uses error validation from FormContext to allow the user to go to the next page and display error messages.

  • FormInputs.js

    • Uses FormContext to display the correct page

  • CustomerInfo.js

    • Uses MUI Textfields. to collect customer information, First Name, Middle Initial, Last Name, Suffix, Address, Email, Phone Number, and Password.

  • DriversLicenseInfo.js

    • Collects the user’s state and driver’s license number using MUI Textfields and Select Menus. Validates the digits in the driver's license number to the state’s standards.

  • CreditCardInfo.js

    • Collects credit card information, and sends to the backend to set up the card using Stripe.

  • ProgressBar.js

    • UI Circles to display current progress and current page.
